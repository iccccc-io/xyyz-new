在 UGC（用户生成内容）社区中，帖子的生命周期远不止“发布”那么简单。用户随时可能**删除、隐藏（设为私密）、编辑（增删改图片和文字）**。如果你的微信数据库状态变了，而 Dify 知识库没跟着变，就会出现严重的**数据泄露或幻觉（AI 搜到了用户不想公开的东西，或者搜到了旧版本的错别字）**。

为了彻底堵死所有逻辑死角，我们需要对 Dify 的 API 进行精准的映射调用。以下是针对这三种高频场景的**终极状态机设计与 Dify 同步规范**，

---

# 需求文档：社区 UGC 帖子生命周期与知识库同步管理规范 

## 1. 核心设计原则
*   **绝对同步**：微信云数据库（`community_posts`）是唯一的“真理源（Source of Truth）”。Dify 知识库必须且只能作为它的**镜像**存在。
*   **软硬结合**：对于用户的“反悔”操作（如设为私密），采用 Dify 的**“禁用 (Disable)”**（软删除）；对于“彻底删除”，采用 Dify 的**“删除 (Delete)”**（硬删除）。
*   **编辑隔离**：每次编辑操作必须重新走一遍完整的安全审核与（必要的）多模态视觉解析管线，绝不允许绕过。

---

## 2. 场景 A：帖子设为私密 / 取消私密 (Visibility Toggle)

**业务定义**：用户将公开的帖子设为“仅自己可见”（`status` 从 `0` 变为 `1`），或者从私密恢复为公开。此时，不仅要在社区信息流中隐藏/显示，**必须立刻在 AI 的知识库中禁用/启用该文档的检索**。

### 2.1 触发与执行逻辑
*   **前置条件**：该帖子必须已存在 `dify_doc_id`。
*   **前端操作**：用户点击“设为私密”或“公开”。
*   **云端流转 (`sync_dify_knowledge` 云函数)**：
    *   读取该帖子的 `dify_doc_id` 和最新的 `status` 状态。
    *   **如果 `status == 1`（私密）**：
        *   调用 Dify API 禁用该文档分段（让 AI 搜不到，但不删向量数据，以便随时恢复）。
        *   **【关键 API 调用】**：目前 Dify 官方 API 暂未提供直接禁用单个 `Document` 的 HTTP 端点。
        *   **【替代方案（推荐）】**：直接调用 **[删除文档]** API (`DELETE /datasets/{dataset_id}/documents/{document_id}`) 将其从知识库抹除。同时清空云数据库中的 `dify_doc_id`。
    *   **如果 `status == 0`（公开，即从私密恢复）**：
        *   系统将其视为**“全新发布”**。
        *   重新走一遍场景 A（发布新帖）的同步逻辑：读取拼装好的 `dify_text`，调用 **[通过文本创建文档]** API，获取新的 `document.id`，并重新写回云数据库的 `dify_doc_id`。

*(注：采用“私密即删除，公开即重建”的逻辑是最稳妥、最不易出错的解法，且 Dify 创建文档速度极快，体验上几乎无损。)*

---

## 3. 场景 B：彻底删除帖子 (Permanent Delete)

**业务定义**：用户或管理员主动删除了帖子记录。不仅要在云数据库中销毁记录，还必须**彻底清理**云存储中的图片/视频，以及 Dify 中的向量碎片。

### 3.1 触发与执行逻辑
*   **云端流转 (`delete_post_pipeline` 云函数)**：
    由于删除动作是破坏性的，必须使用云端事务或 `Promise.all` 确保以下 3 步全部成功：

    1.  **清理 Dify 大脑**：
        *   检查记录中是否存在 `dify_doc_id`。
        *   若有，调用 Dify API：`DELETE /datasets/{dataset_id}/documents/{dify_doc_id}`，返回 `204 No Content` 视为成功。
    2.  **清理云存储（极其重要，防资损）**：
        *   提取帖子 `images` 数组中的所有 `cloud://` 链接。
        *   调用微信云开发的 `wx.cloud.deleteFile` 接口，批量物理删除这些图片，避免占用空间收费。
    3.  **清理数据库**：
        *   最后，从 `community_posts` 集合中彻底 `remove` 该文档。

---

## 4. 场景 C：编辑后重新发布 (Edit & Republish) - 【最复杂逻辑】

**业务定义**：用户修改了帖子的标题、正文文本，或者**增、删、改了部分配图**。由于涉及多模态重新解析，这是极易产生数据脏数据和状态死锁的环节。

### 4.1 前置防线：编辑页的数据结构对比
当用户在编辑页点击“保存/重新发布”时，前端提交的数据可能与数据库原数据存在巨大差异。**云函数必须负责对 `images` 数组进行“差异对比 (Diff)”：**

1.  **对于用户保留的旧图片**：其 `ai_description` 和 `process_status: 1` 必须保留，**无需**重新消耗大模型 Token 解析。
2.  **对于用户删除的旧图片**：将其 `cloud://` 链接收集起来，调用 `wx.cloud.deleteFile` 从云存储中清理掉（防止产生孤儿文件）。
3.  **对于用户新增的图片**：
    *   必须将其在 `images` 数组中的状态初始化为：`ai_description: ""`，`process_status: 0`。

### 4.2 重大状态流转机制 (Pipeline Restart)
完成上述差异对比并更新数据库后，**必须重置整条多模态管线**。

1.  **重置管线状态**：
    *   云函数将该帖子的 `pipeline_status` 强行从 `"synced"` 改回 `"pending"`。
2.  **重新触发视觉解析 (`trigger_ai_pipeline`)**：
    *   由于管线状态变回了 `pending`，系统会自动唤醒 `trigger_ai_pipeline`。
    *   **亮点逻辑**：我们在该函数中写过 `if (img.process_status !== 0) continue;`。这意味着，**它会自动跳过旧图片，仅对刚刚新增的图片发起阿里云 Qwen-VL 的多模态解析**。这极大地节省了 Token 成本！
3.  **最终的 Dify 覆盖同步 (`sync_dify_knowledge`)**：
    *   当所有新增图片解析完毕，管线再次走到同步 Dify 的环节。
    *   此时，云函数必须判断：该帖子是否已经有了 `dify_doc_id`？
    *   **如果有（说明是编辑更新）**：
        *   重新拼装最新的 `dify_text`（包含修改后的文字和新旧图片的 AI 描述）。
        *   调用 Dify API：`POST /datasets/{dataset_id}/documents/{dify_doc_id}/update_by_text`。
        *   **注意**：调用 `update_by_text` 接口时，Dify 会自动删除旧的文档向量并生成新的，无需前端手动先删再建，完美实现覆盖。
    *   **如果没有（说明是全新发布或之前私密过）**：
        *   走 `create_by_text` 接口，并回写新的 ID。

## 5. 开发组自查清单 (Developer Checklist)

为确保以上逻辑无漏洞落地，请开发组核对以下事项：

- [ ] **【私密/公开机制】** 确认采用“私密=调用 DELETE 接口并清空 ID；公开=调用 Create 接口并重写 ID”的方案。
- [ ] **【删除机制】** 确认在删除帖子数据前，已通过 Dify 的 DELETE 接口成功抹除了文档向量，且已调用微信的删除文件接口清除了图片。
- [ ] **【编辑差分对比】** 确认在提交编辑保存时，正确初始化了新增图片的 `process_status=0`。
- [ ] **【管线状态重置】** 确认编辑提交后，将 `pipeline_status` 重置为 `pending`，以激活视觉增量解析。
- [ ] **【Dify 覆盖更新】** 确认同步 Dify 知识库的云函数中，包含通过 `dify_doc_id` 调用 `update_by_text` 进行热更新的代码分支。
