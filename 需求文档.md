这份文档是对整个 **“湘韵遗珍”AI 问一问** 系统的最终集成指南。它将从接口规范、参数透传、到前端组件的渲染逻辑进行**保姆级**的详细拆解，确保开发组能够毫无障碍地完成此次“大版本升级”。

# 研发文档：AI“问一问”系统（V3.0）前端接入与渲染规范

## 1. 升级背景与核心变化
我们已对后端的 Dify 工作流进行了“大厂级”的重构，废弃了原有的散装文本回复模式。
*   **核心变化 1：强制结构化输出**。无论走到哪个逻辑分支（闲聊、科普、社区经验、违规拒答），API **只会且一定会** 返回一个标准的 JSON 对象。
*   **核心变化 2：全场景覆盖**。新增“社区全局搜索”场景，AI 能够跨表检索百科与社区 UGC 帖子，并提供小红书式的“引用溯源”。
*   **核心变化 3：动态上下文注入**。前端必须准确传递用户当前所在的页面环境，作为 AI 意图识别的锚点。

---

## 2. API 通信协议规范 (Dify Integration)

前端仍通过 `wx.cloud.callFunction({ name: 'ai_chat_proxy' })` 进行调用，但入参和出参的数据结构已发生重大变化。

### 2.1 请求参数 (Request Payload)
请在发送给云函数的 `data` 对象中，**严格按照以下结构**传入数据：

```json
{
  "query": "用户的提问字符串",
  "conversation_id": "会话唯一标识 (生成规则见下文)",
  "inputs": {
    "source_scene": "当前页面类型 (枚举值)",
    "source_entity_name": "当前页面主体名称 (如: 湘绣、谭旭生、某篇帖子的标题)",
    "source_entity_id": "当前页面的业务 ID (如: p123, i456)"
  }
}
```

**【关键】：`source_scene` 的枚举值规范（必须完全一致）：**
*   `ich_project`：从非遗项目详情页发起。
*   `ich_inheritor`：从传承人详情页发起。
*   `ich_news`：从非遗资讯详情页发起。
*   `community_post`：从社区单篇帖子详情页/评论区发起。
*   `community_feed`：从社区首页/全局搜索页发起。

**【关键】：`conversation_id` 生成规则：**
为了保证多轮对话的上下文记忆与场景隔离，请在进入聊天页的 `onLoad` 中生成：
`const conversation_id = openid + '_' + source_scene + '_' + source_entity_id;`

---

### 2.2 响应参数 (Response Payload)
后端 Dify 返回的结果，经过云函数代理后，将输出一个**格式绝对固定**的 JSON 对象。请直接读取 `res.result.structured_output`（或云函数中实际对应的解析字段）。

```json
{
  "answer": "湘绣是中国四大名绣之一[1]，许多博主推荐去天心区的体验馆[2]。",
  "citations": [
    {
      "ref_id": "1",
      "title": "湘绣",
      "type": "ich_project",
      "target_id": "07357502cec1..."
    },
    {
      "ref_id": "2",
      "title": "周末探店湘绣馆",
      "type": "community_post",
      "target_id": "b4498fc..."
    }
  ],
  "suggested_questions": [
    "湘绣的针法有哪些？",
    "天心区体验馆多少钱？"
  ]
}
```
*(注：如果是闲聊或拒答，`citations` 为空数组 `[]`。)*

## 3. UI 交互与渲染规范 (Frontend Implementation)

本次升级对聊天窗口的渲染提出了极高的要求，请开发组按以下规范开发一套通用的 `AI_Message_Bubble`（AI 气泡组件）。

### 3.1 “引用溯源”的高级渲染逻辑 (Grounding Render)
当拿到 `answer` 文本时，其中会包含形如 `[1]`, `[2]` 的标号。
1.  **正则匹配**：使用正则表达式（如 `/\[(\d+)\]/g`）提取文本中的标号。
2.  **数据映射**：将提取到的标号 `1`，去 `citations` 数组中查找对应的对象。
3.  **富文本替换**：将标号替换为可点击的蓝色高亮元素（类似小红书）。
    *   *交互要求*：用户点击 `[1]` 时，触发 `bindtap` 事件，根据该标号对应的 `type` 和 `target_id`，使用 `wx.navigateTo` 穿透跳转至对应的百科详情页或帖子详情页。

### 3.2 引用列表平铺 (Citation List) - 可选体验优化
为了让引用来源更显眼，建议在 AI 回复气泡的正下方，渲染一个“参考来源”的横向滚动卡片列表（读取 `citations` 数组）。
*   卡片内容：`[图标(根据type区分)] + [title]`。
*   点击卡片同样触发跳转。

### 3.3 启发式追问 (Suggested Questions)
*   **渲染时机**：当且仅当当前 AI 回复是聊天列表的**最后一条**时，在其下方渲染出 3 个“胶囊按钮”。
*   **数据来源**：遍历 `suggested_questions` 数组。
*   **交互逻辑**：用户点击胶囊按钮，直接将其文本填入输入框并自动发送（触发新一轮请求）。

## 4. 社区模块新增入口需求 (New Entry Point)

除了现有的详情页悬浮按钮，需在**社区全局页面**增加“问一问”的专属入口。

### 4.1 社区首页/搜索页入口设计
*   **位置**：在社区瀑布流的顶部，或搜索框右侧，放置一个紫色的 `[🤖 问一问]` 按钮。
*   **跳转逻辑**：
    *   点击后跳转至 `pages/ai-chat/index`。
    *   携带参数：`source_scene = 'community_feed'`，`source_entity_name = ''`，`source_entity_id = 'global'`。
*   **开场白处理**：针对全局入口，AI 聊天页的初始化开场白应设为：“你好！我是湘韵遗珍生活助手。想了解哪项非遗，或者寻找周末体验攻略？直接问我吧！”。

## 5. 开发组排雷指南 (Checklist)

1.  **JSON 解析容错**：由于我们开启了结构化输出，云端返回的已经是 JS 对象。请开发组在接到数据时，使用解构赋值 `const { answer, citations, suggested_questions } = response_data`，**切勿使用容易报错的纯字符串截取**。
2.  **防抖与异常处理**：
    *   发送请求时必须开启 Loading 状态（气泡显示“正在思考...”）。
    *   如果接口超时或抛出 500 错误，捕获异常并在气泡中显示：“网络开小差了，大师正在重新连线，请稍后再试。”
3.  **空白状态兼容**：如果 `citations` 或 `suggested_questions` 返回空数组，组件必须能优雅降级，仅显示纯文本 `answer`，严禁抛出 undefined 错误。
